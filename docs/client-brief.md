# Project Brief: Podcastr

**Prepared for:** Faith
**Type:** Portfolio Project
**Date:** June 16, 2026

---

## 1. Project Overview

Podcastr is an AI-powered podcast platform where users can both create and consume podcast content. Creators write a text script and the platform generates the audio and a matching thumbnail; the script doubles as the displayed transcript. Listeners get a familiar Spotify-style experience: browse trending and popular shows, search, and stream via a persistent player.

This is being built as a portfolio piece, so the priority is a polished, working demo that's simple to set up and easy to talk through — not enterprise scale. **Setup complexity, not cost or scalability, is the primary thing to manage.**

---

## 2. Goals

- Demonstrate end-to-end full-stack ability: auth, database, file storage, third-party AI APIs, and a polished UI
- Showcase the AI generation loop (script → audio → thumbnail) as the standout feature — and make sure it works end-to-end before polishing anything else
- Ship something deployable and demoable with minimal setup friction
- Keep infrastructure costs at or near zero

---

## 3. Target Users

- **Creators** — want to publish podcast episodes without recording equipment, using AI to generate audio from a script
- **Listeners** — want to browse, search, and stream podcasts with a familiar music-app experience

## 4. Scope (MVP)

### Included

- Sign up / sign in (Google, GitHub, email)
- Home dashboard: Trending, Latest, Popular podcast feeds
- Discover page: search with filters and empty states
- Podcast detail page: playback, description, transcript, similar podcasts
- Create Podcast flow: multi-speaker AI audio generation with per-speaker voice selection, AI or custom thumbnail, category and description
- Podcaster profile: listener count, podcast grid, empty state
- Persistent global audio player
- Sidebar: "Fans Also Like" carousel, Top Podcasters leaderboard
- Edit / Delete own podcasts

### Excluded (post-MVP)

- Comments and ratings
- Follow/subscribe system
- Push notifications
- Creator analytics dashboard
- Native mobile apps

### Scope warning

This is an ambitious feature set for a portfolio timeline (9 screens, full CRUD, AI generation, real-time feeds, a persistent cross-page player). The main risk is a half-finished demo. Build the AI generation loop first and treat vector search as a stretch goal, not a core requirement.

---

## 5. Primary User Flows

1. **Discover & Play** — sign in → browse Home dashboard → open a podcast → play via persistent player → search Discover for more
2. **Create & Publish** — go to Create Podcast → enter title, category, description → write a speaker-labelled script and pick a voice per speaker → generate multi-speaker audio + thumbnail → publish → appears on profile
3. **Manage** — open own podcast → Edit (pre-filled form, re-generates assets) or Delete

---

## 6. Centerpiece Feature: Multi-Speaker Audio Generation

This is the standout feature and the least forgiving part of the build, so the approach is fixed up front.

**Approach: multi-speaker AI narration via Gemini TTS, with user-selected voices.** On the Create page the user sets the number of speakers (1–2 for MVP), writes a script with each line labelled by speaker, and picks a prebuilt Gemini voice for each speaker (with a sample-play preview). Gemini TTS returns a single audio file with distinct, consistent voices per speaker — no chaining separate calls or manually stitching audio. This matches the conversational transcript shown in the original Figma design.

Constraints to engineer around:

- **Preview status** — Gemini TTS is in Preview; expect occasional changes.
- **Length** — quality can drift past a few minutes; input cap is ~8,000 bytes, output ~655 s. Long episodes are chunked server-side and concatenated, preserving per-speaker voices.
- **Intermittent 500s** — the model occasionally returns text instead of audio; the generation server function needs automated retry with backoff.
- **API surface** — use the Gemini API (API key) over the Vertex AI path, which adds GCP project/auth setup.
- **MVP limit** — prebuilt voices only; custom voice cloning is post-MVP.

All AI calls (Gemini TTS, thumbnail, transcription, embeddings) run inside TanStack Start **server functions** (`createServerFn`), so API keys never reach the browser.

---

## 7. Critical Decision: Transcript Source

The transcript should **not** be generated by running speech-to-text over AI-generated audio — that recovers text you already had, at extra cost and with errors introduced.

- **AI-generated episodes:** the transcript _is_ the input script. Store it directly, segmented for display. No Whisper needed.
- **User-uploaded audio** (the Create page also supports custom uploads): use OpenAI Whisper to transcribe, since no script exists.

This split keeps the common path cheap and accurate, and only uses transcription where it's actually required.

## 9. Storage Note (R2 vs Convex)

- **Cost is not a concern either way.** Cloudflare R2's free tier (10 GB storage, generous monthly operations, free egress) far exceeds portfolio needs.
- **R2's one real gotcha:** Cloudflare's CDN cache can mishandle the range requests a browser's audio player sends, causing playback to hang on first load. The fix is a direct public R2 hostname or a small Cloudflare Worker — extra setup that adds little to the portfolio.
- **Simpler alternative:** since the project is already using Convex, Convex's built-in file storage can hold audio and images, removing R2, the Worker, and the range-request problem entirely — one fewer vendor and a cleaner README. The tradeoff is it's less battle-tested for audio range streaming. **Evaluate Convex storage first; fall back to R2 + Worker only if audio playback misbehaves.**

## 10. Search Note (Vector Search)

If vector search is included: embed title + description + transcript as a single combined vector per podcast, truncate long transcripts to fit the embedding model's token limit, and re-embed whenever a podcast is edited.

Be aware that with a small seeded catalog (~10 podcasts), semantic search looks identical to plain text search to anyone viewing the demo. To make it visible, seed enough varied content and prepare demo queries that show off semantic matching. Otherwise, plain text search is the lower-risk MVP choice.
